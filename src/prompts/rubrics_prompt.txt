# TASK
Your task is to build a detailed, dynamic rubrics for evaluating AI responses to user commands in a writing coach system. The system routes user commands to different processing routes based on task type. You must create rubrics that:

1. **Align with the specified metrics** (ONLY include quality dimensions covered by the metrics)
2. **Address all user intents** from the command (explicit and implicit) through the lens of the metrics
3. **Are grounded in the actual input text** (define criteria for relevant, observable changes)
4. **Respect the route's capabilities** (define criteria for what the route is designed to do)
5. **Incorporate the route prompt's guidance** (align with how the agent should behave)
6. **Apply route-specific constraints** (required criteria, exclusions, priorities)

**Critical**: The metrics define WHAT aspects to include in the rubrics. User intents, input text, and route capabilities define HOW to structure the evaluation criteria. Your rubrics must balance all inputs, with metrics serving as the primary filter for criterion selection.

## ROUTING CONTEXT
The system uses an agentic orchestrator that routes commands to one of four routes:

**1. RESEARCH** — Find/explore literature, return papers with citations
- Examples: "Find papers about X", "What does the literature say?"
- Output: Literature search results
- Capabilities: Literature search ONLY (no text editing, no new text generation)

**2. REVISE_RESEARCH** — Edit the manuscript AND search for supporting evidence
- Examples: "Strengthen this with citations", "Add sources to support this claim"
- Output: Text modification suggestions + literature citations
- Capabilities: Text editing + literature search

**3. REVISE_SIMPLE** — Edit without search (grammar, tone, shortening, clarity)
- Examples: "Fix grammar", "Make more concise", "Improve clarity"
- Output: Text modification suggestions
- Capabilities: Text editing ONLY (no literature search)

**4. RESPOND** — Synthesize, discuss, answer meta-questions
- Examples: "Summarize what we discussed", "Compare these approaches", "Explain this concept"
- Output: New text generation
- Capabilities: Generate new content (does NOT modify existing input text)

**Critical Distinction**:
- **REVISE routes**: Suggest modifications to existing text (track changes style)
- **RESPOND route**: Generate new text without modifying input
- **RESEARCH route**: Return literature, no text generation or editing

# RUBRICS DEFINITION
A rubrics is a structured evaluation framework consisting of:
- **Criteria**: Specific aspects of the task to assess (e.g., "Grammar Correction Completeness", "Citation Relevance")
  - Each criterion must map to the specified metric
- **Performance Levels**: Typically 4 levels from Poor (1) to Excellent (4)
- **Indicators**: Observable evidence that would define each performance level

The rubrics you create must be:
- **Metrics-aligned**: All criteria must correspond to specified evaluation metrics; no criteria outside metric scope
- **Specific**: Tailored to the exact user command + input text + route + metrics combination
- **Measurable**: Based on observable, objective indicators that assess the specified metrics
- **Comprehensive within scope**: Cover all aspects of the user's request that align with specified metrics
- **Fair**: Include only what the route is designed to accomplish and what metrics define as important

# INPUT DEFINITIONS

## USER COMMAND
The specific instruction provided by the user that triggered the system response.

**Purpose in Rubrics Generation**:
- **Extract all user intents**: Decompose compound requests (e.g., "fix grammar and improve clarity" = 2 intents)
- **Identify priorities**: Which aspects are primary vs. secondary?
- **Detect implicit requirements**: What must be preserved? (meaning, tone, terminology)
- **Determine success criteria**: What would satisfy the user?

**Examples by Route**:
- RESEARCH: "Find recent papers on climate change mitigation"
- REVISE_RESEARCH: "Add citations to support the methodology claims"
- REVISE_SIMPLE: "Fix grammatical errors and improve sentence flow"
- RESPOND: "Summarize the key differences between qualitative and quantitative methods"

## INPUT TEXT
The original manuscript or text content provided by the user.

**Purpose in Rubrics Generation**:
- **Identify actual issues**: What problems exist that need addressing? (grammar errors, unclear passages, unsupported claims)
- **Understand context**: Domain, document type, writing style
- **Assess scope**: How much work is needed? Localized fixes vs. systemic issues?
- **Determine what to preserve**: What elements should remain unchanged?

**Critical For**: Ensuring rubrics criteria and indicators are grounded in the ACTUAL text provided, not hypothetical issues.

## METRICS DEFINITION
The specific evaluation metrics provided dynamically for this task. Each metric includes a name and description that defines a particular quality dimension to assess. Metrics are the lens through which the AI response will be evaluated when someone uses this rubrics.

**Purpose in Rubrics Generation**:
- **Drive criterion selection**: Only create rubrics criteria that align with the provided metrics
- **Filter irrelevant aspects**: Even if the user command implies certain work, don't include criteria for aspects not covered by the metrics
- **Focus evaluation scope**: Each metric's description defines what aspect of quality matters for evaluation
- **Shape performance indicators**: Design indicators that would measure the specific quality aspects defined by each metric's description

**Important Notes**:
- **Metrics define WHAT to include in rubrics, not HOW to score**: The metrics guide which aspects of the response should be in the rubrics (e.g., correctness, completeness, appropriateness), but the rubrics uses its own 4-level scoring system (Poor/Fair/Good/Excellent)
- **One metric can map to multiple criteria**: A single metric may require several rubrics criteria to fully cover its quality dimension
- **Read metric descriptions carefully**: The description explains the specific aspect of quality to design criteria for

**Critical**: The rubrics should ONLY include criteria for aspects covered by the provided metrics. Don't create criteria for quality dimensions not specified in the metrics, even if they seem relevant to the task.

## ROUTE PROMPT
The system prompt given to the AI agent executing the selected route. This defines:
- **Agent role and capabilities**: What the agent can do
- **Expected behaviors**: How the agent should approach tasks
- **Output format**: Structure and style of responses
- **Quality guidelines**: Standards for good performance
- **Constraints and priorities**: What to emphasize or avoid

**Purpose in Rubrics Generation**:
- **Define the action space**: What types of outputs are within scope?
- **Set behavioral expectations**: How should the agent behave?
- **Identify route-specific quality standards**: What distinguishes excellent from poor performance for this route?
- **Understand design intent**: Why was this route designed this way?

**Example**: A REVISE_SIMPLE route prompt might emphasize "preserve author voice" and "suggest minimal necessary changes", while REVISE_RESEARCH might prioritize "integrate citations smoothly" and "find high-quality sources".

# INSTRUCTIONS FOR RUBRICS GENERATION

Follow this systematic approach to create dynamic, context-aware rubrics:

## Step 1: Decompose the User Command
- **Extract all user intents**: Break down compound requests
  - Example: "fix grammar and improve clarity" → Intent 1: Grammar correction, Intent 2: Clarity enhancement
- **Identify priority**: Which aspects are most critical?
- **Detect implicit requirements**: What does success require even if unstated?
  - Example: "make concise" implicitly requires preserving meaning
- **Note specificity**: Specific ("fix subject-verb agreement") vs. broad ("improve writing")

## Step 2: Identify Evaluation Metrics
- **List all provided metrics**: What quality dimensions are specified in the metrics_definition input?
- **Read each metric description carefully**: What specific aspect of quality does each metric focus on?
- **Understand metric scope in context**: How does each metric apply to this specific user command and input text?
  - A metric about "correctness" might mean grammar correctness for editing tasks, or factual correctness for research tasks
- **Identify coverage boundaries**: What aspects of the task ARE covered by metrics? What aspects are NOT covered (and thus should not be included in the rubrics)?
- **Map metrics to rubrics design**: For each metric, determine what observable aspects should be measured in the rubrics

## Step 3: Analyze the Input Text
- **Document characteristics**: Type, domain, length, style
- **Identify actual issues**: What problems exist that the command should address?
  - Count specific errors if applicable (e.g., "5 grammar errors", "3 unsupported claims")
- **Assess scope**: Localized issues or systemic problems?
- **Note preservation needs**: Elements that must remain unchanged
- **Link to metrics**: Which identified issues are relevant to the specified metrics?

## Step 4: Understand the Route's Action Space
- **Review route prompt**: What is the agent instructed to do?
- **Map route capabilities** to user intent:
  - RESEARCH: Can only search literature
  - REVISE_RESEARCH: Can edit text + search literature
  - REVISE_SIMPLE: Can only edit text
  - RESPOND: Can only generate new text (not edit existing)
- **Identify route-specific quality standards**: What makes a "good" response for this route?
- **Check alignment**: Can this route fulfill the user's request?

## Step 5: Apply Route Rubrics Constraints
- **Include required criteria**: Check if specific aspects must be included in the rubrics
- **Respect exclusions**: Don't include criteria for capabilities the route doesn't have
- **Honor scope boundaries**: Include criteria only for what the route should do
- **Use preferred scoring**: Apply specified scale and interpretation

## Step 6: Map User Intents to Evaluation Criteria (Filtered by Metrics)
For each user intent from Step 1, create corresponding criteria ONLY if they align with the metrics from Step 2:

**Mapping Logic**:
```
User Intent + Relevant Metrics → Evaluation Criterion (or multiple criteria)

Intent: "Fix grammar"
+ Metric focusing on "correctness" → Criterion: "Grammar Correction Accuracy"
+ Metric focusing on "completeness" → Criterion: "Grammar Correction Completeness"
+ NO metric on "impact" → EXCLUDE: "Grammar Improvement Impact"

Intent: "Add citations"
+ Metric focusing on "relevance" → Criterion: "Citation Relevance"
+ Metric focusing on "quality" → Criterion: "Citation Quality"
+ NO metric on "integration" → EXCLUDE: "Citation Integration Smoothness"
```

**Critical Grounding Principles**:
- **Each criterion must map to a specified metric**: Don't create criteria for aspects not covered by the metrics, even if the user intent implies them
- **Metrics determine rubrics scope**: If a metric description focuses on one aspect (e.g., "correctness"), don't include criteria for other aspects (e.g., "improvement impact")
- **One metric may need multiple criteria**: A metric about "quality of suggestions" might require criteria for accuracy, relevance, and appropriateness
- **Ground in inputs**: Each criterion should reference: user intent + relevant metric + input text reality + route capability

## Step 7: Construct Performance Levels (Independent of Metrics Scoring)
For each criterion, define 4 levels with specific, observable indicators that assess the quality dimension described by the relevant metric:

**Critical**: The rubrics uses its own scoring system (Poor=1, Fair=2, Good=3, Excellent=4). The metrics define WHAT to measure, NOT the scoring scale.

**Guidelines**:
- **Align indicators with metric descriptions**: Ensure what you're measuring matches the quality aspect described by the metric
  - If a metric describes "correctness of suggestions", indicators should assess whether suggestions are correct
  - If a metric describes "completeness of coverage", indicators should assess whether all relevant aspects are addressed
  - If a metric describes "appropriateness for context", indicators should assess contextual fit

- **Be specific**: Reference actual input text characteristics
  - ✅ "Corrects 4-5 of the 5 grammatical errors"
  - ❌ "Corrects most errors"

- **Be observable**: Define indicators that a judge can objectively assess based on what the metric describes
  - ✅ "Adds 3-4 citations from peer-reviewed sources"
  - ❌ "Citations are good quality"

- **Be fair**: Align with route capabilities
  - ✅ Include indicators for actions within the route's capability
  - ❌ Don't include indicators for actions the route cannot perform (e.g., don't expect literature search from a route that only edits text)

- **Differentiate clearly**: Each level should be distinct

**Performance Level Structure** (use this 4-level scale for defining rubrics):
- **Excellent (4)**: Define what fully achieving the quality dimension described by the metric looks like
- **Good (3)**: Define what substantially achieving the quality dimension looks like, with minor gaps only
- **Fair (2)**: Define what partially achieving the quality dimension looks like, with significant gaps
- **Poor (1)**: Define what failing to achieve the quality dimension or introducing problems looks like

## Step 8: Validate Rubrics Completeness

Use this checklist:
- [ ] **All provided metrics covered**: Each metric has at least one criterion that addresses its quality dimension
- [ ] **No out-of-scope criteria**: All criteria map to the quality dimensions described by provided metrics
- [ ] **All user intents covered** (within metric scope): Each part of the command has criteria, filtered through metrics
- [ ] **Grounded in input text**: Criteria reference actual issues/characteristics
- [ ] **Route-aligned**: Criteria match route's designed capabilities
- [ ] **Route prompt incorporated**: Agent's instructions reflected in expectations
- [ ] **Constraints respected**: Required criteria included, exclusions honored
- [ ] **Criteria independent**: Minimal overlap between criteria
- [ ] **Indicators measure metric dimensions**: Observable indicators assess what the metric descriptions specify
- [ ] **Levels distinct**: Clear boundaries between performance levels
- [ ] **Scoring independent**: Using 4-level rubrics scoring (1-4), not importing metric scoring
- [ ] **Actionable**: Enables consistent evaluation by different judges

## Step 9: Add Overall Assessment Guidance
- **Prioritize criteria**: Which matter most for THIS specific command/text combination?
- **Note trade-offs**: Any tensions? (e.g., "conciseness" vs. "adding citations")
- **Provide score interpretation**: What ranges indicate satisfactory performance?
- **Add route-specific context**: Special considerations for this route

---

## Key Principles

1. **Metrics-driven focus**: Provided metrics define WHICH quality dimensions to include; rubrics criteria must align with these dimensions
2. **Metrics describe, rubrics score**: Metrics specify WHAT to assess (e.g., "correctness", "completeness"); rubrics provide HOW to score (4-level scale)
3. **Dynamic, not generic**: Rubrics are tailored to the specific user command + input text + route + metrics combination
4. **Route-aware**: Respect different route capabilities and quality standards
5. **Intent-driven (filtered by metrics)**: Every user intent maps to rubrics criteria, but only for dimensions covered by metrics
6. **Grounded**: Reference actual input text issues and characteristics
7. **Observable**: Use concrete, measurable indicators that assess what metrics describe
8. **Fair**: Include criteria for what the route is designed to do and what metrics specify, not what it cannot do
9. **Comprehensive within scope**: Cover all provided metrics and user intents, but nothing beyond the metric scope
