# TASK
Build dynamic rubrics for evaluating an AI writing-coach response. The system routes user commands to different processing routes based on task type. Your rubrics must:

1. **Align with the specified metrics** (ONLY include quality dimensions covered by the metrics)
2. **Address all user intents** from the command (explicit and implicit) through the lens of the metrics
3. **Are grounded in the actual input text** (define rubric items for relevant, observable changes)
4. **Respect the route's capabilities** (define rubric items for what the route is designed to do)
5. **Incorporate the route prompt's guidance** (align with how the agent should behave)
6. **Apply route-specific constraints** (required rubric items, exclusions, priorities)

**Critical**: Metrics define WHAT to evaluate. User intent, input text, route capabilities, and route guidance define HOW each metric-level rubric and its evaluation items should be structured.

**Instruction Priority (highest to lowest)**:
1. Metrics scope and exclusions
2. Route capabilities and route prompt constraints
3. User command intents
4. Input text realities and observable issues
5. Output format requirements

**Output Constraints**:
- Return only the final rubrics in the required format
- Do not include analysis, hidden reasoning, planning notes, or restatements of the instructions
- Create exactly one top-level rubric per provided metric
- The number of top-level rubrics must equal the number of metrics
- Treat every provided metric as high importance
- Group multiple evaluation items under a metric-specific rubric when needed
- Allow evaluation items within a metric to have different importance levels when justified
- Keep each rubric description brief (2-3 sentences)
- Keep each evaluation item description brief (1-2 sentences when possible)
- Keep observable indicators concrete and concise

## ROUTING CONTEXT
The system uses an orchestrator that routes commands to one of four routes:

**1. RESEARCH** — Find/explore literature, return papers with citations
- Examples: "Find papers about X", "What does the literature say?"
- Output: Literature search results
- Capabilities: Literature search ONLY (no text editing, no new text generation)

**2. REVISE_RESEARCH** — Edit the manuscript AND search for supporting evidence
- Examples: "Strengthen this with citations", "Add sources to support this claim"
- Output: Text modification suggestions + literature citations
- Capabilities: Text editing + literature search

**3. REVISE_SIMPLE** — Edit without search (grammar, tone, shortening, clarity)
- Examples: "Fix grammar", "Make more concise", "Improve clarity"
- Output: Text modification suggestions
- Capabilities: Text editing ONLY (no literature search)

**4. RESPOND** — Synthesize, discuss, answer meta-questions
- Examples: "Summarize what we discussed", "Compare these approaches", "Explain this concept"
- Output: New text generation
- Capabilities: Generate new content (does NOT modify existing input text)

**Critical Distinction**:
- **REVISE routes**: Suggest modifications to existing text (track changes style)
- **RESPOND route**: Generate new text without modifying input
- **RESEARCH route**: Return literature, no text generation or editing

# CORE DEFINITIONS
- **Metric-Level Rubric**: One top-level rubric per provided metric. Each top-level rubric must correspond to exactly one metric.
- **Evaluation Items**: Sub-aspects assessed within a metric-level rubric. Each item must stay within the scope of its parent metric.
- **Performance Levels**: Metric-specific levels on a fixed `0-2` scale.
- **Indicators**: Observable evidence that defines each score level.

Your rubrics must be metrics-aligned, specific to the command and input, measurable, grounded, and limited to what the selected route can actually do.

## METRICS DEFINITION
The provided metrics define the evaluation scope.

**Important Notes**:
- Create exactly one top-level rubric per metric
- Keep every top-level metric rubric high priority unless the input explicitly says otherwise
- Use evaluation-item importance only within a metric-level rubric
- Do not add rubric dimensions outside the provided metrics
- Metrics define **WHAT** to evaluate; user intent, input text, and route define **HOW** to structure each rubric

**Current Metrics in Use**:
- **Output Relevancy**: Whether the output or edit suggestions directly address the user command and reflect the user's intent
- **Completeness**: Whether the output covers all required parts of the user's request without omitting critical elements

**Scoring Scheme**:
- **Output Relevancy**: `0 = Irrelevant`, `1 = Partially Relevant`, `2 = Completely Relevant`
- **Completeness**: `0 = Incomplete`, `1 = Adequate`, `2 = Comprehensive`

# RUBRICS GENERATION RULES

Use this workflow:

1. **Extract intents from the user command**: include explicit requests and key implicit requirements such as preservation of meaning, tone, or terminology.
2. **Use metrics as the primary filter**: create exactly one top-level rubric per metric and exclude any dimension not covered by the provided metrics.
3. **Ground the rubrics in the input text**: evaluation items and indicators must be traceable to real issues, claims, constraints, or opportunities in the input text.
4. **Respect route scope**: only include expectations the selected route can satisfy.
5. **Map intents into evaluation items under each metric**: group related sub-aspects under the relevant metric-level rubric instead of creating extra top-level rubrics.
6. **Use the fixed scoring scheme**:
   - **Output Relevancy**: `0 = Irrelevant`, `1 = Partially Relevant`, `2 = Completely Relevant`
   - **Completeness**: `0 = Incomplete`, `1 = Adequate`, `2 = Comprehensive`
7. **Make indicators observable**: use concrete evidence and thresholds where possible.

**Compact mapping logic**:
- If the user asks to fix grammar, **Output Relevancy** should focus on whether the edits target the requested grammar issues, while **Completeness** should focus on whether all major requested issues are covered.
- If the user asks to add citations, **Output Relevancy** should focus on whether the sources support the targeted claims, while **Completeness** should focus on whether all major unsupported claims within scope are covered.
- If a quality dimension is not in the provided metrics, exclude it even if it seems useful.

Checklist before finalizing:
- Every provided metric has exactly one top-level rubric
- All user intents are covered within metric scope
- No rubric or item exceeds route capabilities
- All indicators are grounded, distinct, and measurable
- The score levels clearly match the fixed `0-2` metric-specific definitions
- Final output matches the required format exactly
